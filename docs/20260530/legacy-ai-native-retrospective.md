# 이전 프로젝트에서 AI-Native하게 적용 가능했을 것으로 보이는 업무들

> 스터디 1회차 후속 정리 · 2026-05-30 · 정리 kodayoung \
> 성격: 이전 프로젝트 회고 / 발표 후 추가 논의 \
> 관점: 단계별 AI-Native 적용 가능성 · trade-off 중심

---

## 1. 분석 단계

### 적용 가능 업무
- 화면별 연계 SP / Table 추출
- SP 기준 metadata 및 의존성 분석
- 호출 관계 및 데이터 흐름 추적
- 정책 후보 자동 추출
- 운영 이력 기반 Edge Case 후보 도출
- 사장된 코드 / 실사용 코드 구분 보조
- hidden dependency 탐지 보조

### 기대 가능 효과
- 초기 온보딩 및 분석 기간 단축
- 레거시 구조 파악 비용 감소
- 추가 검토 필요 항목 자동 도출
- 구조 및 흐름 레포팅 자동화

### 현실적 한계
- SP만으로 정책 확정 불가능
- DB metadata / 운영 흐름 / 샘플 데이터 기반 추가 검증 필요
- 동적 SQL 및 운영 배치 기반 hidden dependency 존재 가능

---

## 2. 설계 단계

### 적용 가능 업무
- 리스크 도출
- 아키텍처 대안 비교
- ADR(Architecture Decision Record) 초안 작성
- 검증 전략 및 테스트 범위 제안
- 정책 흐름 기반 구조 후보 제안

### 기대 가능 효과
- 설계 근거 정리 자동화
- 리스크 분석 보조
- 설계 문서화 비용 감소

### 현실적 한계
- 구조 설계 자체를 AI가 대신하기는 어려움
- 회사별 운영 정책 및 조직 구조 반영 필요
- 실제 활용성은 검증 경험 부족

---

## 3. 개발 단계

### 적용 가능 업무
- PRD/SRS 표준화 기반 코드 생성
- CRUD 및 반복 코드 자동 생성
- 정책 기반 테스트 코드 생성
- Edge Case 기반 테스트 자동화
- 코드 리뷰 자동화

### 기대 가능 효과
- 반복 구현 비용 감소
- 테스트 범위 확대
- 리뷰 품질 편차 감소

### 현실적 한계
- 구조 안정성 및 명세 표준화 선행 필요
- 정책 변경 흐름 관리 필요
- 최종 코드 검증 및 운영 책임은 사람 영역

---

## 4. 검증 단계

### 적용 가능 업무
- ERP 비교 결과 Diff 분석
- 차이 유형 분류
- 유사 패턴 클러스터링
- 원인 후보 추론
- 버그 수정 프롬프트 생성

### 기대 가능 효과
- 대규모 검증 데이터 분석 시간 단축
- 반복 검증 비용 감소
- 운영 이슈 추적 보조

### 현실적 한계
- 최종 정산 계산 검증 자체는 deterministic한 테스트 필요
- AI는 검증자가 아니라 분석 보조 역할에 가까움
- 차이 원인 최종 확정은 사람 검증 필요

---

## 5. 운영 / 협업 단계

### 적용 가능 업무
- 정책서 RAG 기반 온보딩/문의 챗봇
- 정책 변경 시 관련 문서 변경안 자동 생성
- Rule 기반 사장 문서 탐지 및 최신화 Alert
- 조직 간 용어 / 정책 Context 정리

### 기대 가능 효과
- 운영 문의 감소
- 정책 전파 속도 향상
- 문서 최신화 비용 감소
- 온보딩 비용 감소

### 현실적 한계
- 프로젝트 초기 단계에서는 문서 및 정책 정리 부족
- 초기부터 RAG 기반 구조 적용 어려움 예상
- 운영 안정화 이후 적용 가능성 높음
- 자동 삭제/자동 승인 구조는 위험

---

## 현실적인 트레이드오프

### 필요 선행 작업
- Context 정리
- 문서 표준화
- metadata 연결
- 검증 체계 구축

### 예상 비용
- 초기 AI-Native 구조 셋업 비용 증가
- 문서 및 정책 정리 비용 증가
- Context 관리 비용 증가

### 핵심 판단
- 단순히 AI 모델 연결만으로 해결되지는 않음
- AI가 안정적으로 이해할 수 있는 구조 자체를 먼저 만드는 비용 필요
